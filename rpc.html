JSchema-RPC

Introduction

To complement JSchema, JSchema-RPC is a way to specify a RPC end point using JSON.

RPC URL Convention and File Conventions

Assuming an end point is at 

http://example.com/path/
then the JSchema-RPC should be available at 
http://example.com/path/?JSchema-RPC

Locally, JSchema-RPC files should end with the .jscrpc file suffix.

Grammar

The grammar for a JSchema-RPC file builds on the grammar for JSchema and JSON: 

<jschema_rpc_doc> ::= 
  '{'
    '"description"' ':' <string> ',' 
    '"url"' ':' <string> ',' 
    <optional_typedefs>
    '"functions"' ':' '[' <function_specs> ']'
  '}'
  
<optional_typedefs>
  '' |
  <typedefs_map>

<function_specs> ::= 
  <function_spec> | 
  <function_spec> ',' <function_specs> 

<function_spec> ::= 
  '{' '"name"' ':' <string>,
      '"description"' ':' <string>,
      '"args"' ':' '[' '{' <arg_spec> '}' ']' ','
      '"return_type"' ':' <type> 
  '}'
  
<arg_spec> ::=
  <string> ':' <type> <default> <description>

<default> ::=
  ',' 'default' ':' <object> |
  ''
<description> ::=
  ',' 'description' ':' <string> |
  ''

Interpretation

Given a JSchema-RPC Document and a function specification, an invocation of the function will be done as follows:

  • The "name" of the function will be appended to the "url" (or a different URL if the user has specified one)
  • The arguments will be passed along in a manner appropriate for the scheme specified in this composed URL. In the case of http and its variants, this will be either as a query string or as post variables as a GET or POST, respectively. Values must be provided for every parameter that does not have a default value. Note: if an end point does not support a given HTTP method, it should say so in its documentation.
  • The return content will conform to the type specified as the "return_type" unless an exception is returned, in which case it will conform to the grammar specified in the "Exceptions" section.

Exceptions

Exceptions may be raised by a JSchema-RPC end point using a return value of the following form: 

<jschema_rpc_exception ::= 
  '{'
    '"exception@"' ':' <string> <optional_trace>
  '}'

<optional_trace ::=  
  '' | 
  ',' '"trace@"' ':' <string>
End point creators should strive to provide useful traces, including elements only relevant to the remote procedure, rather than implementation details of the invocation.

Tools Note

It is expected that JSchema-RPC documents will be downloaded into local code repositories to avoid development-time dependencies on network resources. When this is done, tool writers should make an effort to follow the transitive closure of JSchema resources and bring them down to the local machine, with the appropriate updates to references, ideally using relative URLs.

Ideally it should be easy for developers to update these local resources as necessary.

Examples

JSchema-RPC File Example usage (Gosu)
{ "description" : "Methods for manipulating employees", "url" : "http://myserver:8080/employees", "typedefs@" : { "Employee" : { "first_name" : "string", "last_name" : "string", "age" : "integer", "id" : "biginteger" } } "functions" : [ { "name" : "getEmployee", "description" : "Returns the employee of the given name", "args" : [ {"id" : "biginteger" } ], "return_type" : "Employee" }, { "name" : "updateEmployee", "description" : "Updates the given employee", "args" : [ { "employee", "Employee" } ], "return_type" : "boolean" } ] } // Assuming the above file is in EmployeeController.jscrpc: var empController = new EmployeeController() var myEmp = empController.getEmployee(42) myEmp.age++ if( empController.updateEmployee(myEmp) ) { print( "Updated the age of ${myEmp.first_name}") }
